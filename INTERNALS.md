# INTERNALS
ultra-css-template と css-api の、コードだけからは読み取れない設計判断・数値根拠のメモ。「なぜこの値なのか」「なぜこの構造なのか」を残す。

## 定数の根拠

### `BASE = 316781`（info の width 上限 + 桁上げの底）
info チャンネルは値を SVG の寸法にエンコードし、CSS 側が scroll-timeline で読み取ってデコードする。このデコードは

```
round( 1 / round_sig6( 1 / (n + 1) ) - 1 ) = n
```

という形になる。CSS の計算は有効数字 6 桁で丸められる（`round_sig6`）ため、`n` が大きくなるとこの等式が成立しなくなる。`n = 1, 2, 3, …` と探索したとき、**初めて左辺が `n` に一致しなくなるのが `n = 316781`**。したがって width で正しく往復できる上限がこの値であり、桁上げの底として採用している。

### height の上限が 350
height にはステータスも詰め込んでおり（`900 × floor(value / BASE) + (status − 100)`）、`floor(value / BASE)` の取りうる段数が上限になる。`BASE = 316781` に対して

```
350 × 900 + 899 < 316781 < 351 × 900 + 899
```

が成り立つため、height 側の段数上限は 350 となる。

### `VALUE_MAX = 999999`（s の上限）
これも CSS の有効数字 6 桁に由来する。`1000000` を `calc()` などの計算関数（`min` のような選択型を含む、あらゆる計算関数）に通すと、表記が `1e+06` に変形される。

- `<number>` として使う分には `1e+06` は `1000000` と等しく、これ以下で計算している限り何の問題もない。
- しかし `<integer>` 文脈で使うと、IACVT とも別の「汚染状態」になり、**この値およびこれを利用した値がすべて無効値に伝染していく**。

汚染の影響範囲は、プロパティに直接突っ込む分にはほぼ影響なし。ただし `if()` の `style()` で判定したり、`@function` の引数に渡したりした時点で破壊される。これを避けるため、6 桁に収まる `999999` を上限にしている。

## `<integer>` 型宣言の挙動
`@property` で `<integer>` を宣言したカスタムプロパティに値を入れたときの挙動は 4 通り。

1. **`<number>` として無効な値** → `initial-value` になる。値の参照が正常な箇所に飛ぶだけなので、汚染ではなく普通に計算できる。
2. **直接の整数 `n` を指定** → `clamp(-2147483648, n, 2147483647)` にクランプされる。
3. **丸め可能な数値関数**（`round(|x|) < 1000000` となる `calc` / `min` / `clamp` 等すべて）を使った数 → 丸められる（最大で有効数字 6 桁の精度に潰れる）。
4. **丸め無効な数値関数**（`round(|x|) ≥ 1000000`）を使った数 → **汚染状態**。プロパティの構文自体は正常なので `initial-value` に落ちず、さらにこの値を使用したプロパティへ汚染を伝搬させる。通常の `<integer>` のそぶりをしているため、あらゆる型のカスタムプロパティを汚染・伝染させられる。

`VALUE_MAX = 999999` は、この 4 番目（汚染）を踏まないための上限。

## api-getter の `--api-ok()` に余裕を持たせている理由
`--api-ok()` の成功判定を `200 ≤ status` ではなく `197 ≤ status < 300` と少し広げているのは、digital-clock のように連続 animate でデコードする経路だと、値が 3600 付近で少しぶれて status が `200` から `197` などに落ちることがあるため（加えて 19x 番台には何もステータスを登録していないのも理由の一つ）。整数を返す通常の info API ではきっちり `200` で読めるので、そちらではほぼ問題にならない。

## digital-clock：height を離散・width を連続にした理由

- **width（連続 animate）**：時刻の連続的な進行を表す。ただし小数になるため、整数前提の api-getter では値が大きくなるほど小数状態での誤差で震える。この震えは 3600 程度までなら許容範囲。60 段階を離散で書くのは綺麗でないので、width は連続にしている。
- **height（24 段階の離散 animate）**：height は「時」を表し 24 段階しかないので離散化し、小数状態そのものをなくして誤差を消している。もう一つの理由として、height はステータスも詰め込んでいるため、段数は 24 でも値としては `900 × 24` まで大きくなる。これを連続値にすると width よりずっと大きい値が小数で震えることになり壊れる。

## pulse-tap：37.5ms `allow-discrete` のフレーム設計

`pulse-tap` の transition に使う `37.5ms step-end allow-discrete` は、判定単位が 37.5ms（= 8.33333ms × 4.5）になるように選んである。

- これを利用する animation の 1 段階分を 50ms（= 8.33333ms × 6）にしておく。
- 判定に固定で 1 フレームを消費する。
- `4.5` はちょうど 4 と 5 の中間なので、安定して 5 フレーム目に繰り越される（ここを整数にするとブレる）。
- `5 + 1 = 6` で animation の 6 フレームと揃い、**37.5ms 以上の長押しがタップ判定になって、自動でちょうど 1 段階分変わる**。

**前提**：この設計は 1 フレーム（8.33333ms ≒ 120fps）が安定して刻まれること、つまりフレーム割れが起きないことに依存している。ページが重くてフレーム落ちが発生すると、フレーム数の勘定（固定 1 フレーム消費・5 フレーム目への繰り越し・6 フレームとの整合）がずれ、判定が崩れる。正確に動作するのはページがある程度軽い状態に限られる。

## macbook-keyboard：counter-style だけでキーを組む
実際の MacBook のキーボードを実測し、その寸法をもとに `min-width` / `text-indent` / `letter-spacing` / `word-spacing` / `tab-size` / `max-width` の 6 値で連立不等式・連立方程式を組んで各キーの見た目を作っている。キーの見た目を同時に成立させるうえで邪魔になる `letter-spacing` は最終的に外している。

### オフセットの内訳
各記号が各プロパティに与える幅（monospace 前提）：

| 記号 | 効かせる先 | monospace での幅 |
| --- | --- | --- |
| `''`（空） | min-width | `0px` |
| `' '`（半角空白） | text-indent / letter-spacing / word-spacing | `1ch` |
| `'　'`（全角空白） | text-indent / letter-spacing | `1ic`（= `2ch`） |
| `'0'`（通常文字） | — | `1ch` |

`letter-spacing: -1ic` のとき：`''`→`0px`、`' '`→`-1ch`、`'　'`→`0px`、`'0'`→`-1ch`。
`word-spacing: 1ch` のとき：`''`→`0px`、`' '`→`0px`、`'　'`→`0px`、`'0'`→`-1ch`。

### 連立の解
各プロパティを

```
min-width      = a
text-indent    = b
letter-spacing = c
word-spacing   = d
tab-size       = e
```

とし、各記号が生む合計幅を測ると：

```
''    : a           = A
' '   : b + c + d   = B
'　'  : b + c       = C
' 　' : b + 2c + d  = D
'\9'  : e           = E
```

制約（各記号幅が min-width 以上、`c ≥ 0`、など）を整理すると：

```
解：
  a = A
  b = B + C − D
  c = D − B
  d = B − C
  e = E

制約：
  A ≤ (B, C, D, E)
  (B, C) ≤ D
  B + C − D (= b) < E
```

### 任意 6 数への拡張（max-width を追加）
`max-width = f (= F)` を足し、`'  '`（全角空白 2 つ）の幅が F 以上になる条件を加える：

```
(A, B, C, D, E) ≤ F
'  ' : b + 2c + 2d ≥ F
→ (D, E) ≤ F ≤ b + 2c + 2d
```

代入して整理すると：

```
(D, E) ≤ F ≤ B + D − C
D ≤ B + D − C  ⟹  C ≤ B
B + C − D < E ≤ B + D − C  ⟹  C < D

解：
  a = A,  b = B + C − D,  c = D − B,  d = B − C,  e = E,  f = F
制約：
  A ≤ (B, C, D, E) ≤ F ≤ B + D − C
  C ≤ B ≤ D
  B + C − D (= b) < E
```

### font-size を 0 にする別解
`font-size` を 0 にすると、このオフセット調整は不要になる（monospace を指定する必要もなくなる）。font-size 比例のタブ文字の暗黙幅もほぼ 0（0.000003px 程度）になるので、基本的にはこちらの方がよい。ただし注意点として、height を指定しないと高さが 0 になる、`lh` や `em` などフォントサイズ基準で他の余白（margin 等）を調整していた場合はそこが崩れる。

## todo-list：contenteditable × shadow DOM による複製とフィルタ

### 複製の仕組みと破壊経路の封じ込め
`contenteditable` が DOM を複製する挙動を利用し、shadow DOM ごと複製させることで構成要素を丸ごと複製している。Enter で複製できるが、複製はカーソルが先頭にある時でないと安定しない。そのため、以下の破壊経路を順に潰している。

- 1 文字だけ入れた状態で、→/↓ キー → Enter、→/↓ キー → Backspace、連打で単語選択してから Backspace、といった操作で壊れる。
- 対策として、次のカーソル位置が描画領域の外にある時に →/↓ キーを押すと自動スクロールが発生することを利用し、`letter-spacing` を枠サイズの 2 倍より大きくしておくことで、枠内のどこを押しても 1 カーソル目にカーソルが来るようにする。
  - `word-spacing` ではなく `letter-spacing` を選んだのは、そうすることで →/↓ キーだけでなく `a` などの文字キーを押した場合でも先頭維持を効かせるため。
- スクロール状態に応じて `user-select: none` にし、どのキー入力も受け付けないようにする。同時に overlay を枠より右側の width 全体を埋めるように配置して支配し、押すとスクロール状態を先頭にリセットして、次に押しても大丈夫にする。
- 連打で単語選択状態になり Backspace で消せてしまうのを防ぐため、クリックしたら overlay の button を「連打モード解除の標準秒数」である 500ms 間だけ出す。次にクリックすると button 判定に吸わせ、デフォルトの単語選択時間 500ms をリセットして単語選択を不可能にする。

これにより、delete で消す・Command+A で全選択してから Backspace、の 2 手段以外の破壊経路をすべて塞いでいる。

### フィルタリングの仕組み
shadow DOM では `:host(...)` のように「その要素単体」に対するセレクタしか外から引き継げない。一方 CSS 変数は shadow DOM を貫通する。この性質を使い、**セレクタ → CSS 変数 → shadow DOM 貫通 → 内部でその CSS 変数による出し分け**という流れでフィルタを制御している。

## dice-cube：navigate 属性の由来
各 index を、サイコロの展開図の対応する番号に割り当てておくことで、`::scroll-marker` の標準機能によりクリックした面を正面に向かせられる。navigate は「body 主体のスクロール」をさせる属性として作ったので、クリック時に押した面が正面を向く。これが navigate という命名の理由。
